### 交换机（Switch）是工作在**数据链路层**的网络设备，主要功能是**根据 MAC 地址转发数据帧**。

| 设备 | 工作层级 | 寻址依据 | 作用 |
|------|---------|---------|------|
| 集线器 Hub | 物理层 | 无 | 广播所有数据 |
| 交换机 Switch | 数据链路层 | MAC 地址 | 根据 MAC 转发 |
| 路由器 Router | 网络层 | IP 地址 | 根据 IP 路由 |

## VLAN端口类型

### Access 端口（接入端口）
- 属于**一个** VLAN
- 发送出去的帧**不带** VLAN 标签
- 连接：电脑、服务器、打印机等终端设备

### Trunk 端口（干道端口）
- 可以承载**多个** VLAN
- 发送出去的帧**带** VLAN 标签（除了 PVID 对应的 VLAN）
- 连接：交换机和交换机之间

### Hybrid 端口（华为特有）
- 灵活配置哪些 VLAN 带标签，哪些不带
- 可以同时属于多个 VLAN
- 华为设备常用

## VLAN 基础

### 什么是 VLAN？
**VLAN（Virtual LAN）**：虚拟局域网，把一个物理交换机逻辑上分成多个虚拟局域网。

### VLAN 的作用
1. **隔离广播域**：不同 VLAN 不能直接通信，减少广播
2. **提高安全性**：不同部门/不同 VLAN 隔离
3. **灵活组网**：不受物理位置限制

### VLAN 范围
| VLAN ID | 范围 | 说明 |
|---------|------|------|
| 0 | 保留 | 不能用 |
| 1 | 默认 VLAN | 所有端口默认在 VLAN1 |
| 2 ~ 1001 | 标准 VLAN | 正常使用 |
| 1002 ~ 1005 | 保留 | Token Ring / FDDI |
| 1006 ~ 4094 | 扩展 VLAN | 高级应用 |

### 802.1Q 标签
- 在源 MAC 地址后面插入 **4 字节**的 VLAN 标签
- TPID：2字节，固定 0x8100
- TCI：2字节，其中 12 位是 VLAN ID（所以最多 4096 个 VLAN）
- PVID：端口默认 VLAN ID

## 七、VLAN 数据转发过程
PC1（VLAN10）→ Access口（加标签VLAN10）→ Trunk口（带标签转发） → 对端Trunk口（识别标签）→ Access口（去掉标签VLAN10）→ PC2（VLAN10）

**规则**：
- Access 端口**收**：不带标签 → 加 PVID 标签
- Access 端口**发**：带标签且等于 PVID → 去掉标签发出
- Trunk 端口**收**：有标签 → 保留；无标签 → 加 PVID 标签
- Trunk 端口**发**：VLAN 在允许列表中 → 如果等于 PVID 则去掉标签，否则带标签

## VLAN 间路由

不同 VLAN 默认**不能直接通信**，需要三层设备：

### 方式1：单臂路由
- 路由器一个物理接口，划分多个子接口
- 每个子接口对应一个 VLAN
- 缺点：单点故障、带宽瓶颈

### 方式2：三层交换机（推荐）
- 通过 VLANIF 接口（SVI）实现
- 性能高，是现在主流方案

## 反思

**Q1：交换机和集线器的区别？**
- 集线器：物理层，广播所有数据，所有端口在同一冲突域
- 交换机：数据链路层，根据 MAC 转发，每个端口一个冲突域

**Q2：MAC 地址表的老化时间？**
- 默认 300 秒（5分钟）

**Q3：交换机的三大功能？**
- 学习、转发、泛洪

**Q4：Access 和 Trunk 的区别？**
- Access：一个VLAN，不带标签，接终端
- Trunk：多个VLAN，带标签，接交换机

**Q5：VLAN 的作用？**
- 隔离广播域、提高安全性、灵活组网

**Q6：不同 VLAN 怎么通信？**
- 通过三层设备：路由器（单臂路由）或三层交换机（VLANIF）

## 一、RDS 是什么？

**RDS（Relational Database Service）**：关系型数据库服务，阿里云提供的托管数据库服务。

### RDS vs 自建数据库对比
| 对比项 | 自建数据库（ECS上装MySQL） | RDS 云数据库 |
|--------|------------------------|-------------|
| 安装维护 | 自己装，自己升级补丁 | 托管，自动升级 |
| 备份恢复 | 自己写脚本 | 一键备份，一键恢复 |
| 高可用 | 自己做主从 | 自带主备高可用 |
| 监控告警 | 自己搭监控 | 自带监控告警 |
| 扩容 | 麻烦，要停机 | 弹性扩容，几分钟搞定 |
| 成本 | 服务器成本 + 人力成本 | 按需付费，省人力 |

## 三、RDS 核心概念

### 高可用
- 主备架构：一主一备，自动故障切换
- 多可用区部署：跨机房容灾

### 备份恢复
- 自动备份：每天自动备份
- 手动备份：随时手动备份
- 按时间点恢复：可以恢复到任意时间点
- 备份保留：默认保留 7 天

### 监控
- CPU、内存、磁盘、连接数
- 慢查询日志
- 告警设置

---

## 四、安全最佳实践

1. **用内网地址连接**：不要用公网地址，速度快又安全
2. **白名单精细化**：只允许需要的IP访问，不要全开 0.0.0.0/0
3. **强密码**：数据库账号密码要复杂
4. **最小权限**：应用账号只给需要的权限
5. **定期备份**：开启自动备份，定期手动备份

## 五、反思

**Q1：为什么用 RDS 而不是自己装 MySQL？**
- 不用自己维护（升级、补丁、备份）
- 自带高可用（主备自动切换）
- 弹性扩容方便
- 自带监控告警
- 省人力成本

**Q2：RDS 连接不上怎么排查？**
1. 检查白名单有没有加对
2. 检查 ECS 和 RDS 是否在同一个 VPC
3. 检查安全组（RDS 是白名单机制，不是安全组）
4. 检查用户名密码是否正确
5. 检查端口 3306 是否通

**Q3：RDS 怎么备份？**
- 自动备份：设置备份策略，每天自动备份
- 手动备份：随时手动创建备份
- 可以按时间点恢复，也可以按备份集恢复

**Q4：RDS 可以从公网访问吗？**
- 可以，但不推荐
- 需要开通外网地址，安全性差
- 生产环境建议只用内网地址
