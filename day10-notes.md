# VLAN间通信和阿里云云监控配置钉钉告警实验

## 📚HCIA学习
   VLAN 间通信

## 🎯 实验目标
基于阿里云云监控 2.0 主机监控，为两台 ECS 实例配置 CPU 使用率告警规则；当 CPU 平均值≥80% 且持续 3 个周期时，同时向绑定手机发送短信、向钉钉群推送 IM 告警消息，实现双渠道运维告警通知。

## 📌 实验简介
本次实验基于阿里云云监控服务，完成钉钉机器人告警渠道配置、全局通知策略绑定，通过压测模拟 ECS CPU 高负载，验证 ECS、RDS 云资源异常可自动推送告警至钉钉群，实现云上资源实时监控预警能力。

## ⚙️ 实验环境
- 云平台：阿里云
- 可用地域：华东1（杭州）
- 实验资源：两台 ECS 服务器、一台 RDS MySQL 数据库
- 辅助工具：stress 压力测试工具、钉钉自定义机器人

## 📝 完整操作流程
### 1. 🤖 钉钉群创建自定义 IM 机器人
目标钉钉群 → 群设置 → 智能群助手 → 添加自定义机器人；
安全设置勾选「自定义关键词」，仅填写单个关键词：告警，不添加任何标点、不开启 IP 白名单、不加签；
保存机器人，复制完整 Webhook 地址备用。
### 2. 📊 ECS 批量安装云监控采集 Agent
云监控控制台左侧「通知对象」下方切换至【主机监控】；
勾选两台 ECS 实例，点击「批量安装 / 升级插件」；
等待 3 分钟完成安装，刷新页面，确认 CPU 使用率出现数值，指标正常采集。
### 3. ⚙️ 创建 CPU 使用率告警规则
左侧菜单栏【报警规则】→【新建报警规则】，关联两台 ECS 实例；
添加监控指标：简单指标 - 主机 CPU 使用率，平均值≥80%，连续 3 个 1 分钟周期；
必填报警联系人组选择系统自带「云账号报警联系人」，实现短信推送；
展开下方高级设置，在报警回调输入框粘贴钉钉机器人完整 Webhook 地址；
右侧测试按钮即便返回关键词不匹配报错也无需处理，真实告警自带关键词可正常推送；
弹性伸缩、日志服务、函数计算等推送渠道全部关闭，恢复通知保持绿色开启；
页面底部点击【确认】保存完整告警规则。
### 4. 🧪 ECS CPU 压测验证告警链路
SSH 远程登录 ECS 执行压测命令：
bash
运行
stress --cpu $(nproc)

保持终端窗口持续运行 10 分钟以上，等待 3 个完整采集周期触发告警；
云账号绑定手机收到 CPU 过载告警短信；
钉钉群同步接收云监控告警消息。

## ❌ 实验问题与解决
### 故障 1：新建联系人组无法搜索到钉钉 IM 机器人
现象：联系人分组左侧列表搜索机器人显示 Not Found，无法移入联系人组；
根因：阿里云云监控联系人组仅支持手机号 / 邮箱账号联系人，IM 机器人属于独立 Webhook 回调通道，二者分属两套推送体系，无法混合加入；
解决方案：放弃将机器人加入联系人组，采用「联系人组短信 + 独立 Webhook 钉钉回调」双渠道并行推送，互不干扰。
### 故障 2：报警回调测试返回 errcode:310000 关键词不匹配
现象：测试模板报文提示关键词不匹配，但真实 ECS 告警可正常推送钉钉；
根因：云监控内置测试模板为简易模拟报文，未携带「告警」关键词；真实业务 ECS 告警报文正文固定包含「告警」二字，满足钉钉机器人匹配规则；
解决方案：测试报错无需修改机器人配置，直接保存告警规则，压测产生真实告警事件后钉钉自动接收消息。
### 故障 3：短信、钉钉完全收不到任何告警通知
根因：ECS 未部署云监控采集 Agent，无 CPU 监控指标数据，云监控不会生成告警事件，所有推送渠道全部失效；
解决方案：主机监控页面批量安装采集插件，等待指标稳定采集后再执行压测验证。

## 📚 实验收获
云监控告警分为两套独立推送链路：联系人组负责短信 / 邮件，Webhook 回调负责钉钉 IM 消息，可同时启用，实现多渠道冗余告警；
钉钉自定义机器人关键词校验机制仅对真实业务告警生效，云监控内置测试模板不具备参考价值，测试报错不代表实际推送失败；
主机监控依赖采集 Agent 上报指标，无监控数据则所有告警渠道均失效，采集组件部署是告警生效前置条件；
运维告警配置多渠道并行推送，单一推送链路拦截时，短信渠道可保障告警通知可达性，提升运维可靠性。
## ✅ 实验验证结果
ECS 长时间持续 CPU 压测触发告警阈值后，手机短信、钉钉群双渠道同步收到 CPU 高负载告警通知，全部实验目标达成。
